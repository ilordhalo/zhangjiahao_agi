Developer Guide · Runtime v0.4.0

从 Linear Issue 到可审阅、可恢复的实现报告

Symphonz 轮询 Linear，把符合条件的 Issue 交给独立 Codex app-server 进程；Runtime 持久化任务、事件、错误和结构化报告，并用认证 Dashboard 提供重启后仍可访问的审核入口。

标准库 RuntimeSQLite 持久历史结构化报告工具LAN 单用户认证
1. Poll 与路由项目、状态、标签、依赖和 attempt 预算共同决定 eligibility
2. Workspace 与 Codex复用 issue 目录和分支，每次 invocation 建立 app-server thread
3. Review 与报告Agent 创建 PR/MR，Runtime 验证、保存、渲染并同步报告评论
4. Human Review 与清理人工决定 Rework/Merging；终态删除 workspace，但历史与 artifact 保留

02 · Linear Polling

轮询结果先过 eligibility，再按状态选择工程路由

SymphonzPoll 分页读取 active_states；Orchestrator 再检查标签、阻塞依赖、并发 claim、终态和持久 attempt 预算。Linear 是外部控制面，状态变化就是下一次 Agent 指令。

1

加载 tracker 配置

读取 project slug、API key 环境变量名、active/terminal states、required labels。

2

分页执行 SymphonzPoll

每页最多 50 条；缺少 end cursor 或 GraphQL errors 会形成结构化 tracker error。

3

检查 eligibility

排除缺标签、被未完成 Issue 阻塞、已 claimed、终态或超过 max_attempts 的记录。

4

按状态路由

Todo/In Progress 实现，Ready to Publish 发布，Rework 返工，Merging 合并。

5

并发与退避

不同 Issue 受 max_concurrent_agents 限制；失败进入有预算的指数退避，预算跨重启保留。

eligibility.routeorchestrator.py
candidate = project + active_state + labels

eligible = (
  not claimed
  and dependencies_completed
  and attempt_budget_available
  and state not in terminal_states
)

Ready to Publish     -> publish report + review
Human Review         -> inactive pause

重启语义：attempts.sqlite3 保存状态级预算，runtime.sqlite3 保存可查询历史。服务重启不会把失败任务伪装成第一次执行，也不依赖进程内最近 200 条事件恢复 Dashboard。
03 · Codex Protocol

每次 worker invocation 都建立明确的 app-server 生命周期

Runtime 在 Issue Workspace 中启动独立子进程，完成 initialize 握手、initialized 通知、thread/start 和一个或多个 turn/start。所有工具调用、事件和退出都经过同一 JSON-RPC 流。

逐步点击可查看触发条件与持久化效果。
#LinearRuntimeWorkspaceCodexStore
完整链路

从 Linear 候选查询开始，到持久报告、人工审核门和终态清理结束。

进程边界：每次 invocation 创建一个 app-server 进程和 thread；同一 invocation 的 continuation turn 复用 thread。结束、取消或 timeout 时 Runtime 关闭整个进程组，避免遗留 child process。
04 · WORKFLOW Anatomy

当前 WORKFLOW.md 为什么这样写

WORKFLOW.md 把“何时触发”和“触发后必须完成什么”放在同一个可版本化文件中。Front matter 是 Runtime 的配置层，正文是完整发送给 Codex 的 Prompt 层；Runtime 不假装理解项目工程步骤，Agent 也不能接管认证、报告存储或清理。

配置层 · Runtime 读取

tracker 查询、workspace hooks、并发/重试预算、Codex command、approval policy 和 sandbox 都是确定性运行参数。

Prompt 层 · Agent 执行

Issue context、状态路由、Workpad 格式、分支/Review 约定、验证、报告发布和 Human Review 门是工程行为约束。

1. tracker 与 polling

声明 Linear project、active/terminal states 和标签。Linear 是外部控制面，Human Review 故意不在 active_states。

Runtime-owned

2. workspace 与 hooks

一个 identifier 对应一个可复用目录；首次 clone，轮次前后检查，终态执行 before_remove 后安全删除。

Runtime-owned

3. Codex 与 sandbox

固定 app-server command、timeout、workspace-write 和 networkAccess；Runtime 传递边界并负责进程组退出。

Runtime-owned

4. Issue context

每个 turn 都注入 identifier、title、state、labels、URL、description 和 attempt，允许从 Linear 与文件系统恢复。

Runtime-owned

5. Operating Rules

限制工作目录、禁止虚构成功、要求验证和同步，使 unattended session 有明确停止条件。

Agent-owned

6. 单一 Workpad

一个 Issue 只维护一个 Workpad。它记录计划、验收、验证、branch、review 和 report URL，跨 turn 保留 Agent 上下文。

Agent-owned

7. Status Map

Todo 实现、Ready to Publish 发布、Rework 修改、Merging 合并。Ready to Publish 是发布命令，Done 是终态。

Agent-owned

8. Branch 与 Review

确定性 issue branch、目标分支、PR/MR 标题和复用规则减少并发 turn 对 provider 状态的歧义。

Agent-owned

9. 发布门与 guardrails

报告必须在 review request 存在之后发布；报告同步 pending 时不得进入 Human Review。Runtime-owned 报告评论与 Agent-owned Workpad 分离。

Runtime + Agent
05 · Issue Lifecycle

状态既是队列，也是 Codex 指令选择器

Runtime 只调度活动状态；Agent 按 Prompt 中的状态表执行。Human Review 是人工审核门，不是后台轮询 review comments 的状态。

Todo

claim 后先转 In Progress，建立 Workpad 与 issue branch。

In Progress

实现、测试、commit；仍活动时 continuation/retry。

Ready to Publish

push、创建 PR/MR、调用 symphonz_report、等待 sync。

Human Review

暂停 Codex，由人决定 Rework 或 Merging。

Merging

更新 base、检查 CI、merge 成功后转 Done。

Done

Human Reviewreviewer 要求修改 →Rework复用 branch/report comment →Human Review；Cancelled / Canceled / Duplicate 直接进入终态清理。
06 · Persistence & Reporting

报告由 Agent 提供结构，Runtime 决定安全形态和同步状态

symphonz_report 只接受有界 JSON 字段。ReportPublisher 校验 Issue 身份、URL、节点/边和证据，转义所有文本，原子写入版本化 JSON/HTML，再让 RuntimeStore 切换权威 generation。

07 · Dashboard Auth

Dashboard 读取持久历史，并用单用户 LAN session 保护每条业务路由

symphonz install 或 symphonz configure-dashboard 写入非 secret 的 [dashboard] 与 mode 0600 的 auth.toml。除 /login 和 /healthz 外，HTML、report 和 API 都先认证。

1. 配置与 KDF

优先 scrypt-v1；不可用时使用 600,000 次 PBKDF2-SHA256。salt、password hash、session secret 不写 config.toml。

2. 登录限流

稳定 64 个 client bucket，每个请求先获得 SQLite reservation，再执行 KDF；第五次失败锁定 15 分钟。

3. Session cookie

浏览器拿随机 token，数据库只存 SHA-256 digest。Cookie 使用 HttpOnly、SameSite=Lax、Path=/ 和 Max-Age。

4. 重启与轮换

session 存在 RuntimeStore，服务重启后仍有效；username 或 session secret 轮换会使旧 session 失效，logout 删除记录。

0.4.0 安全边界这是 trusted LAN 单用户认证，不是公网身份系统。HTTP 请求会持续显示“未加密连接”提示；公网部署、TLS termination、SSO、注册、角色和找回密码均不在本版本范围。
08 · Human Review Gate

Human Review 把自动执行和最终决策明确分开

Agent 只有在 branch、review request、验证、报告 artifact 和 Linear 报告评论全部同步后，才可把 Issue 从 Ready to Publish 移入 Human Review。

进入门前

Review request 已存在；symphonz_report 返回稳定 URL；Runtime-owned 评论包含可点击 report URL 与经 neutralization 的 review target；sync state 是 synced。

门内暂停

Human Review 不触发 Codex。Reviewer 在 GitHub/GitLab 审查，外部人员或自动化把 Linear 改为 Rework 或 Merging。

门后路由

Rework 复用 branch、Workpad、review 和同一报告评论；Merging 重新检查 base 与 CI，成功 merge 后才写 Done。

11 · Implementation Boundaries

阅读运行结果时必须保留的边界

这些约束决定 0.4.0 的恢复、安全和审核语义。

01

RuntimeState 不是历史数据库

进程内集合服务当前调度；Dashboard 的 task/timeline/error/report 查询来自 RuntimeStore。

02

completed 不等于 Linear Done

turn completed 只表示协议正常结束；业务终态必须由 Linear 与 provider 外部事实确认。

03

Workpad 与报告评论分离

Agent 更新工作过程；Runtime 幂等维护报告链接。重试不能覆盖或复制另一方的评论。

04

pending 报告可用但不可交审

本地 artifact 和 URL 已存在时，Linear 暂时失败仍返回 pending；后台重试，直到 synced 才过发布门。

05

artifact 不随 workspace 删除

报告 bundle、SQLite 和日志位于 workspace 之外，Done/Closed/Cancelled 清理不破坏审核历史。

06

认证是 LAN 边界

单用户 session、KDF 和限流不等于公网 TLS、反向代理、SSO 或多租户授权。

07

Agent 不能提交可执行报告

symphonz_report 接收结构化字段；Runtime 转义文本并使用固定渲染器，避免同源脚本和报告漂移。

08

失败必须可查询

失败 event 写结构化 error 与 errors.jsonl；恢复操作设置 resolved_at 和 resolving_event，不抹除历史。
