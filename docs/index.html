Developer Guide · Current Implementation

一条 Linear Issue，如何变成一次 Codex 工程任务

Symphonz 是连接任务系统和编码代理的轻量调度层。它持续监听 Linear，将符合条件的 Issue 放进独立工作目录，再通过 Codex app-server 启动一轮带完整项目规则的开发会话。

默认每 5 秒轮询 每个 Issue 独立 Workspace JSON-RPC 触发 Codex
01
监听 Linear按项目、状态和标签查询候选 Issue
02
准备 Workspace创建目录，首次进入时克隆目标仓库
03
触发 Codex渲染 WORKFLOW Prompt，启动 thread 和 turn
04
发布与审核Codex 根据 Issue 状态推送分支并创建 PR / MR

02 · Linear Polling

Linear 不是发 Webhook，而是由服务周期性拉取

run_service() 按 polling.interval_ms 进入循环。每轮查询指定项目中处于活动状态的 Issue，再由 Orchestrator 做标签和运行中去重。

1

加载工作流配置

读取项目 slug、API key 引用、活动状态、必需标签与轮询间隔。

2

执行 SymphonzPoll

向 Linear GraphQL 发送查询，一次最多获取 50 条候选 Issue。

3

筛选候选 Issue

GraphQL 已按项目和状态过滤；本地再确认 required_labels。

4

跳过正在运行的 Issue

若 identifier 已存在于 state.running，本轮不重复启动。

5

同步执行并等待

当前实现逐条调用 run_issue()，一个 Codex turn 完成后才处理下一条。

6

等待下一轮

默认休眠 5000ms，最小强制为 1000ms，然后重新查询 Linear。

SymphonzPoll.graphqllinear.py
query SymphonzPoll(
  $projectSlug: String!,
  $stateNames: [String!]!,
  $first: Int!
) {
  issues(filter: {
    project: { slugId: { eq: $projectSlug } },
    state: { name: { in: $stateNames } }
  }, first: $first) {
    nodes {
      id identifier title description priority
      state { name }
      branchName url
      labels { nodes { name } }
    }
  }
}
当前调度模型：Orchestrator 按优先级领取候选 Issue，过滤未完成 blocker，并依据 agent.max_concurrent_agents 进行有界并发执行。同一 Issue 的 claim 保证每轮只会进入一个 worker。
03 · Codex Trigger

每个 Issue 如何触发一次 Codex app-server turn

Symphonz 不调用一次性的 codex exec。它启动配置中的 app-server 命令，通过逐行 JSON-RPC 建立客户端、创建 thread、启动 turn，并持续消费事件直到结束。

04 · WORKFLOW Anatomy

当前 WORKFLOW.md 为什么这样写

WORKFLOW.md 同时承担两种职责：Front matter 是 Symphonz Runtime 读取的配置层，正文是每次完整发送给 Codex 的 Prompt 层。把两层放在一个文件中，可以让“何时触发”和“触发后怎么做”保持同一个版本。

配置层 · Runtime 读取

定义 Linear 查询条件、轮询、workspace hook 和 Codex app-server 启动参数。Python 服务只解析 front matter，不执行正文里的 Git/Linear 操作步骤。

Prompt 层 · Codex 执行

把 Issue 上下文、状态路由、Workpad 格式、分支约定、验证和发布规则作为一次完整工程任务发送给 Codex。

1. Tracker 与活动状态

tracker:
  api_key: $LINEAR_API_KEY
  project_slug: "..."
  active_states:
    - Todo
    - In Progress
    - Ready to Publish
    - Merging
    - Rework
设计目的

密钥只写环境变量引用，避免安装生成的文件落入真实 secret。active_states 不是“未完成列表”，而是可触发 Codex 的命令集合：Todo/In Progress 做开发，Ready to Publish 做发布，Rework 做返工，Merging 做合并。

Linear 是外部控制面：人或外部自动化通过改变 Issue 状态选择下一种执行路径。Human Review 被刻意排除，因此审核期间不会持续消耗 Codex。

Runtime 查询人改变状态

Runtime 使用线程安全 claim 去重；若 Issue 后续仍处于活动状态，会在下一轮继续同一 workspace，并携带递增的 attempt 上下文。

2. 轮询、Workspace 与首次 Hook

polling:
  interval_ms: 5000
workspace:
  root: .symphonz/workspace
hooks:
  after_create: |
    git clone --depth 1 "$SYMPHONZ_REPO_URL" .
设计目的

轮询让接入方不需要部署 Webhook endpoint。每个 Issue 使用固定目录，使不同任务隔离，同时让后续 turn 能看到前一轮留下的分支、commit 和测试结果。

after_create 只在目录第一次创建时执行，避免每轮重新 clone 覆盖未发布改动；之后直接复用现有 workspace。

Runtime 创建目录Runtime 执行 Hook

任务按 max_concurrent_agents 有界并发执行；同一 Issue 的 workspace 生命周期 hook 仍保持顺序。

3. Codex app-server 与 Sandbox

codex:
  command: codex ... app-server
  approval_policy: never
  thread_sandbox: workspace-write
  turn_sandbox_policy:
    type: workspaceWrite
    networkAccess: true
设计目的

app-server 提供明确的 initialize → thread/start → turn/start 协议和事件流。无人值守场景使用 approval_policy: never，避免 turn 等待无法回应的确认框。

写权限限制在 Issue workspace；网络访问用于 Linear、Git provider、依赖下载和 CI 查询。权限边界由 Runtime 传给 Codex，但工具凭据仍由运行环境提供。

5. Operating Rules

1. Unattended session; do not ask a human.
2. Work only inside the workspace.
3. Keep one persistent Workpad.
4. Ticket Validation/Test Plan is required.
5. Reproduce before changing bug behavior.
6. Keep Linear, branch and review state synced.
设计目的

无人值守不是降低标准，而是把原本依赖人工提醒的工程纪律写进 Prompt：限制目录、先复现、遵守验收条件、持续同步外部状态，并在缺少权限时留下可诊断的 blocker。

7. 状态映射就是执行路由

Todo             → start implementation
In Progress      → continue workspace
Ready to Publish → publish branch + PR/MR
Human Review     → pause
Rework           → address feedback
Merging          → validate + merge
Done             → terminal
设计目的

Ready to Publish 是发布命令，Done 是终态。这样“实现完成”“准备发布”和“已经安全进入主分支”不会混为一谈。Human Review 是明确的人机边界，Rework 与 Merging 则由外部状态变化重新唤醒 Runtime。

人选择路由Codex 执行路由

8. 确定性分支与 Review Request

branch:
  symphonz/{{ issue.identifier }}-<slug>
title:
  {{ issue.identifier }}: {{ issue.title }}
target:
  SYMPHONZ_MR_TARGET → base branch → main
设计目的

稳定命名让后续 turn 能找到同一分支和开放中的 PR/MR。若 review request 仍开放就复用，避免重复创建；若已关闭或合并，才从最新主分支重新开始。

Codex 创建/复用人审核

9. Guardrails 与阻塞处理

Never push unvalidated changes.
Never expand scope silently.
Publish only from Ready to Publish.
Treat Done as terminal.
Do not merge with failing checks.
On missing provider auth:
  record blocker → Human Review
设计目的

这些规则把高风险操作变成显式门槛。网络、凭据或 provider 工具缺失时，Codex 保留本地 commit、更新 Workpad 并转 Human Review，而不是丢失工作或伪造成功。

Codex 停止高风险动作人处理 blocker
05 · Issue Lifecycle

Linear 状态既是队列，也是 Codex 的执行路由

Runtime 负责把活动状态 Issue 交给 Codex；具体应该实现、发布、返工还是合并，由 WORKFLOW.md 中的状态规则决定。特别注意：Ready to Publish 是发布触发器，Done 是终态。

Todo

进入队列；Codex 先转到 In Progress。

In Progress

在既有 workspace 内实现并验证。

Ready to Publish

推送 issue 分支并创建 PR / MR。

Human Review

暂停自动调度，等待人工决定下一状态。

Merging

审核通过后，同步 main、确认 CI、执行 merge。

Done

合并完成后的正常终态，并清理 workspace。

Human Review反馈需要修改 →Rework完成后返回 →Human Review
任意阶段终止 →Cancelled / Canceled / Duplicate
默认暂停点：Human Review 默认不在 active_states，Runtime 不会主动轮询 PR / MR 评论。需要人工或外部自动化把 Issue 改为 Rework 或 Merging，才会再次触发 Codex。

Symphonz Runtime 直接负责

运行循环与 Codex 会话的基础设施职责。

  • Linear GraphQL 查询
  • Issue 标签筛选
  • Workspace 创建与 after_create hook
  • Prompt 渲染与 Codex JSON-RPC
  • 运行事件和 Dashboard 状态

Codex 按 WORKFLOW.md 负责

需要项目凭据、工具和业务判断的外部操作。

  • 更新 Linear 状态和 Workpad
  • 修改代码、运行测试、创建 commit
  • 推送 issue 分支
  • 创建 GitHub PR 或 GitLab MR
  • 在 Rework 处理已提供的反馈
  • 在 Merging 检查 CI 并执行 merge

Human Review 不触发 Codex。这个最简四轮路径假设负责人在 Turn 1 完成后把 Issue 在下一次 poll 前改为 Ready to Publish；若它继续停留在 In Progress，实际运行可能出现额外的 In Progress turn。后续也只有外部把 Issue 改为 Rework 或 Merging，下一轮 Linear poll 才会创建新的 Codex thread 和 turn。

每一轮共同携带的固定 Prompt 框架

Runtime 实际传入的是完整 WORKFLOW.md 正文，不会只发送下面某一条状态路由。每个案例 Prompt 块展示的是“本轮变化的 Issue context + 从完整规则中生效的指令”。固定部分始终包括：

  • 无人值守执行，不要求人完成正常步骤
  • 只能修改当前 Issue workspace
  • 只维护一个 ## Symphonz Workpad
  • Issue Validation/Test Plan 是强制验收条件
  • Bug 必须先复现或检查现状
  • Linear、Workpad、branch、commit、PR/MR 必须同步
  • 缺少权限时记录 blocker 并进入 Human Review
  • 状态映射与 Todo/In Progress/Ready to Publish/Rework/Merging 步骤
  • 固定 branch、PR/MR title 和 target 约定
  • 未经验证不得 push，检查失败不得 merge

07 · Project Layout

安装后的项目结构与启动方式

symphonz install 把工作流配置写入当前项目。运行时本身来自全局安装的 Symphonz CLI，不再要求项目 clone 外部 Symphony 或安装 mise / Elixir。

project/
├── .symphonz/
│   ├── WORKFLOW.md       # 配置与 Codex Prompt
│   ├── config.toml       # 安装信息、环境变量引用
│   ├── logs/             # 服务日志目录
│   └── workspace/
│       ├── ZHA-8/        # 一个 Issue 一个工作目录
│       └── ZHA-9/
└── ...                   # 原项目文件
symphonz install交互式收集 Linear 与 Git provider 配置，并创建 .symphonz。
symphonz run仅启动 Linear 监听和 Codex 调度，不启动 Web Dashboard。
symphonz run --port 4100同时在 127.0.0.1:4100 提供 Runtime Dashboard。
symphonz version查看当前 CLI 版本；本介绍页对应 v0.3.0。
08 · Implementation Boundaries

阅读运行结果时，需要知道的八个当前边界

这些不是工作流愿景，而是 v0.3.0 内置 Runtime 的真实行为。明确它们，才能正确理解 Dashboard 和 Linear 中看到的状态。

01

状态保存在进程内存

RuntimeState 没有持久化存储；服务重启后，running、completed、blocked 和事件历史会从空状态重新开始。

02

有界并发执行

Orchestrator 依据 max_concurrent_agents 并发处理不同 Issue，并通过 claim 防止同一 Issue 重复执行。

03

completed 不等于 Linear Done

Dashboard 的 completed 表示当前 Codex turn 正常结束；Linear Issue 可能仍是 Todo、In Progress、Ready to Publish、Rework 或 Merging。

04

活动 Issue 会继续推进

若 Issue 下一轮仍属于 active_states，Runtime 会复用 workspace，并用 attempt 和 Workpad 恢复上下文。

05

发布依赖 Codex 环境

Git push、GitHub PR、GitLab MR 与 Linear 状态更新依赖 Codex 会话中可用的 token、CLI 或 MCP 工具。

06

终态负责自动清理

Runtime 会单独查询终态，取消仍在执行的 worker，运行 before_remove，再安全删除对应 workspace。

07

指数退避重试

失败 turn 会进入 retrying，按 attempt 指数退避，并受 max_retry_backoff_ms 和 max_turns 限制。

08

结构化运行日志

Runtime 同时维护进程内存状态，并把事件追加写入 .symphonz/logs/runtime.jsonl，便于故障追踪。

工程结论Symphonz 当前是一个轻量、可观察的轮询与会话调度器；项目级开发流程的智能部分由 Codex 在 WORKFLOW Prompt 约束下完成。
